## Context

GitBench has a working Astro site (`gitbench/web/`) with dark-themed pages, Recharts-based visualizations, and a shadcn/ui component kit. The site reads a single `results.json` file generated by Python's `render.py aggregate_runs()`. React components are loaded as `client:load` islands.

Five polish items were identified through usage: model selectors on the same page don't sync, pricing exists in the OpenRouter pipeline but doesn't reach the UI, the Explore tag cloud is overwhelming, a mock "unknown" model pollutes every view, and the site lacks introductory/methodology content.

The pricing discovery spike confirmed that OpenRouter returns `usage.cost` (float USD) and `usage.cost_details` in every response body. The OpenAI Python SDK v2 stores extra fields in `usage.model_extra`, accessible as plain attributes. The `Score` dataclass already has `cost_usd` and `provider_cost_usd` fields — they're just never populated.

## Goals / Non-Goals

**Goals:**
- All ModelSelector instances on the same page share identical selection state
- Model selection persists across page navigations via localStorage
- `cost_usd` flows from OpenRouter API call → Score → results.json → TypeScript types → UI display
- Token counts (`input_tokens`, `output_tokens`) are visible on model output cards and model overview
- A cost-vs-quality quadrant chart helps users identify value-efficient models at a glance
- The Explore tag interface is compact and searchable rather than a wall of pills
- The `"unknown"` model is filtered at aggregation so it never appears in any view
- Dashboard is renamed Overview with intro text; a Methodology page explains the benchmark process

**Non-Goals:**
- Changing how benchmarks run or score (only the output pipeline)
- Adding pricing to ModelSelector badges (deferred per user direction)
- Real-time or live data updates (static JSON remains the data source)
- Changing the Compare page's `?with=` parameter behavior
- Adding pricing data for Ollama/local models (they don't return cost)
- Internationalization or i18n of intro/methodology content

## Decisions

### D1: ModelSelector sync via localStorage + CustomEvent

**Choice:** Store selected model names as JSON array in `localStorage` key `gitbench-model-selection`. On mount, each `ModelSelector` reads this key (falling back to all models selected if absent). On change, write to localStorage and dispatch `new CustomEvent('model-selection-changed', { detail: newSelection })` on `window`. Each instance listens for this event and updates its internal state.

**Alternatives considered:**
- **React Context**: Would only work within a single provider tree. Charts are sibling islands, not parent-child — each is a separate React root. Context doesn't bridge island boundaries.
- **URL query params**: Would pollute URLs and require re-reading on every route change. Doesn't help same-page sync since query params change the page.
- **`window.storage` event alone**: Only fires in *other* tabs, not the current page. Insufficient for same-page sync.

**Why not `BroadcastChannel`:** Overkill. `BroadcastChannel` is for cross-origin/cross-tab messaging. localStorage events already handle cross-tab; CustomEvents handle same-page. The combination covers all cases with simpler APIs.

### D2: Cost extraction in model.py

**Choice:** In `OpenAIAdapter.generate()`, after extracting `usage`, also extract `cost = getattr(raw_usage, 'cost', None)` and include it in the returned usage dict. Use `getattr` because `cost` lives in `model_extra` — Pydantic surfaces it as a regular attribute but it won't appear in `__dict__`.

**Why this location:** The `generate()` method already builds the usage dict and is the single point where the raw OpenAI SDK response is accessed. Adding cost extraction here requires no changes to the return type or caller interface.

### D3: Cost threading through Score → aggregation

**Choice:** In `runner.py`'s `_run_fixture()`, map `usage.get("cost")` onto `score.cost_usd`. In `render.py`'s `aggregate_runs()`, when building the per-fixture dict, add `"cost_usd": score.get("cost_usd")`. In `web/src/lib/types.ts`, add `cost_usd: number | null` to `FixtureResult`.

**Why no `provider_cost_usd` passthrough:** The `provider_cost_usd` field on `Score` was likely intended for the OpenRouter `provider` field (e.g., "SiliconFlow"), but that's a string, not a cost. We'll leave `provider_cost_usd` unset for now — it can be populated later if a use case arises.

### D4: Quadrant chart implementation

**Choice:** A new `CostValueChart.tsx` React component. Use Recharts `ScatterChart` with two additional `ReferenceLine` components at the median X (cost) and median Y (pass rate). Each model is a dot. Hover tooltip shows model name, pass rate, avg cost. Click navigates to model detail.

**Data computation:** Client-side aggregation from the loaded data: for each model, sum `total_tokens` across all fixtures, multiply by per-model pricing if available, divide by fixture count to get average cost. Alternatively, pre-compute model-level cost aggregates during `aggregate_runs()` and store in `model_summaries`.

**Preferred approach:** Compute in `aggregate_runs()` to keep the chart component simple. Add `avg_cost_usd` and `total_cost_usd` to each model's summary dict. This avoids re-computation in every consumer.

### D5: Tag cloud → search-first interface

**Choice:** Replace the flex-wrap tag cloud with:
1. A prominent `<input type="search">` for tag filtering (autocomplete from the full tag list)
2. Below: a compact row of top 8-10 tags as clickable pills ("Popular tags")
3. A "Show all N tags" toggle that expands the full list in a grid/collapsible
4. Existing filter selects (difficulty, benchmark) remain below

**Implementation:** Pure client-side JS in the existing `<script>` block on `explore.astro`. No new React component needed — it's a progressive enhancement of the existing HTML structure.

**Why not a D3 force layout or treemap:** The data has no hierarchy. A force-directed word cloud is visually impressive but takes more code to build than the value it provides for what is fundamentally a filtering UI.

### D6: Unknown model filtering

**Choice:** Filter in `aggregate_runs()` after all computation. Before returning the result dict, remove `"unknown"` from `model_list`, `model_summaries`, `matrix`, `fixtures`, and `runs_meta`. This is ~15 lines and is the last operation before `return`.

**Why not filter earlier:** Filtering during the aggregation loop would make the code harder to follow. Doing it at the end as a cleanup pass keeps the aggregation logic unchanged and the filter obvious.

### D7: Methodology page architecture

**Choice:** A new `pages/methodology.astro` with static content (no data dependency). The page uses `Layout` with title "Methodology". Content is hardcoded Markdown-style prose in the Astro template. No React islands needed — it's a pure content page.

**Navigation placement:** Add to the sidebar's `links` array as the last item with a `BookOpen` icon from lucide-react. No divider separator to keep the sidebar clean.

## Risks / Trade-offs

- **[Risk] localStorage quota**: If model names are very long and numerous, localStorage has a 5-10MB limit. → Mitigation: Array of model names is at most a few KB. Not a realistic concern.
- **[Risk] CustomEvent listener leak**: If ModelSelector components mount/unmount rapidly, event listeners could pile up. → Mitigation: Use `useEffect` cleanup to `removeEventListener` on unmount.
- **[Risk] Cost computation inaccuracy**: If some fixtures lack token data (e.g., errors), per-model average cost may be skewed. → Mitigation: Only include fixtures with valid `total_tokens > 0` in cost averages. Display "—" for models with no cost data.
- **[Risk] Unknown model data loss**: Filtering `"unknown"` at aggregation means it's gone from `results.json`. If someone needs that data for debugging, it's only in raw run files. → Mitigation: Document this in the Methodology page's "Data availability" section. Raw run files in `gitbench-results/` remain untouched.
